# .brain — Boutique Platform code wiki

> A 1:1 wiki of this repository. Every tracked file has exactly one page. The LLM is the
> librarian: it reads the working tree, compiles it into structured interlinked pages, and
> keeps them honest over time.

This vault follows the LLM Wiki pattern from the `second-brain` skill family, with a set of
**documented deviations** required to make an article-shaped schema work for source code.
Read `## Deviations From Canonical Schema` before writing any page. Canonical schema:
`~/.agents/skills/second-brain/references/wiki-schema.md`.

---

## The one rule that matters most

**The brain is authoritative for orientation. The code is authoritative for behavior.**

- *Orientation* — what a file is for, what depends on it, which concepts it participates in,
  where its tests are, and which other files you need to open.
- *Behavior* — signatures, defaults, values, control flow, error paths, SQL, config.

Never state a behavioral fact on the strength of a page alone. Open the file.

## Architecture

```
.brain/
├── CLAUDE.md      this file
├── raw/           NO copies — the raw corpus is the git working tree (see raw/README.md)
├── output/        scan and drift reports (gitignored)
├── queue/         machine-local session state (gitignored)
├── hooks/         SessionStart / SessionEnd shell hooks
├── scripts/       brain-page-path.sh · brain-scan.sh · brain-index.sh
├── templates/     page-{code,doc,config,generated,vendored,symlink,directory}.md
└── wiki/
    ├── index.md   GENERATED by brain-index.sh — never hand-edit
    ├── log.md     append-only; one entry per wave/sync; never edit existing entries
    ├── sources/   mirrors the repo tree — one page per tracked file, plus _index.md per dir
    ├── entities/  third-party tools, services, orgs (FastAPI, SQLAlchemy, uv, pnpm, qmd…)
    ├── concepts/  cross-file abstractions, patterns, DB tables (Row Level Security…)
    └── synthesis/ comparisons and cross-cutting analyses (hand-authored)
```

## Page path is a pure function of repo path

```
.brain/wiki/sources/<repo-relative-path>.md
```

`backend/app/auth/service.py` → `.brain/wiki/sources/backend/app/auth/service.py.md`

The extension is retained deliberately. Never reimplement this mapping — call
`bash .brain/scripts/brain-page-path.sh <repo-path>`.

**Path casing trap.** Git tracks `backend/` and `frontend/` in lowercase even though the
on-disk directories are `Backend/` and `Frontend/`. macOS case-insensitivity hides this, but
`git ls-files Backend` returns nothing. All `path:` fields and all pathspecs use lowercase.

## Frontmatter — required on every page

```yaml
---
tags: [backend, auth, python]
sources: [backend/app/auth/service.py]
created: YYYY-MM-DD
updated: YYYY-MM-DD
# --- .brain extensions (see § Deviations) ---
path: backend/app/auth/service.py
blob: <git hash-object of the file at authoring time>
commit: <git rev-parse HEAD at authoring time>
kind: code
applicability: active
---
```

| Field | Values |
|---|---|
| `path` | repo-relative, lowercase as git stores it. Omitted on concept/entity/synthesis pages. |
| `blob` | `git hash-object <path>`. **The staleness key.** |
| `commit` | `git rev-parse HEAD`. Used only to *explain* drift, never to detect it. |
| `kind` | `code` · `doc` · `config` · `generated` · `vendored` · `symlink` · `directory` · `concept` · `entity` · `synthesis` |
| `applicability` | `active` · `vendored-inapplicable` · `dead` · `n/a` |

There is **no `status:` field**. Status is computed by `brain-scan.sh`, never stored — a
stored status is a second thing that can go stale.

Directory `_index.md` pages use `blob = git ls-files -s <dir> | shasum` so that adding or
removing a file in a directory marks the directory page stale.

## Granularity

- The **file** is the atomic page unit. Nothing smaller gets its own page.
- **Sub-file symbols** are `##`/`###` anchors inside the file page, cited as
  `[[backend/app/auth/service.py#authenticate]]`.
- The **directory** is the module unit → one `_index.md` per directory.
- **Cross-file abstractions and DB tables** → `wiki/concepts/`.
- **Third-party tools** → `wiki/entities/`.

## Operations

**Ingest** — use `/brain-ingest <dir>`, not `/second-brain-ingest`. One confirmation per
batch (a directory, max 15 files), not per file. `/second-brain-ingest` remains correct for
genuinely external material dropped into `.brain/raw/`.

**Sync** — use `/brain-sync`. Recomputes missing/stale/orphan from git, rewrites stale pages
from their diffs, regenerates the index, appends one log entry.

**Query** — read `wiki/index.md`, follow the tree to a directory `_index.md`, then to leaf
pages. For large searches use `qmd query "…" -c <collection>`. Note `qmd search --path` is
**silently ignored** — always use `-c`.

**Lint** — `bash .brain/scripts/brain-scan.sh --lint` for structural checks;
`/second-brain-lint` for the semantic checks (contradictions, orphans) it does well.

## Writing rules

1. Never modify a repo file to make a page true. The code wins; fix the page.
2. `wiki/index.md` is generated. Never hand-edit it. Run `brain-index.sh`.
3. `wiki/log.md` is append-only. One entry per wave or sync, written by the orchestrator only.
4. Every reference to a **tracked** repo file is a wikilink to its exact path:
   `[[backend/app/auth/service.py]]`. Bare unlinked paths in prose are forbidden.
   Paths to **untracked or ignored** files (`backend/.venv/`, `frontend/node_modules/`) go in
   backticks — no page exists for them.
5. Every page carries all frontmatter fields above.
6. When a page contradicts the code, the code is right. Update the page and note what changed.
7. Never record line numbers. Cite symbols by name.
8. Prefer updating an existing page over creating a new one.
9. Never create a symlink from inside `.brain/` to anywhere under `/Users/mrwen/second-brain/`
   — the vault already symlinks *into* here, and a link back would form a traversal loop.
10. Parallel writers touch leaf pages only. `wiki/index.md`, `wiki/log.md`, and every
    `_index.md` are orchestrator-owned.

## Deviations From Canonical Schema

The canonical schema says "You never improvise structure — you follow these rules exactly."
These deviations are therefore written policy, not drift. Each names the stock rule it
replaces and why.

**1. Rule 4, "Never use raw file paths in page content" — OVERRIDDEN.**
For a code wiki the file path *is* the primary key. Because source page filenames mirror repo
paths, `[[backend/app/auth/service.py]]` is simultaneously a valid wikilink and the literal
path, so the rule is honored in spirit: references are still links, never bare strings.

**2. Title Case wikilinks and kebab-case filenames — SPLIT BY BUCKET.**
`sources/` pages use the verbatim repo path as filename stem, `# H1`, and wikilink text — no
Title Case, no slugification, no stripping of `_ . /`. Slugifying would destroy exactly the
characters that make identifiers unique. `entities/`, `concepts/`, and `synthesis/` keep stock
Title Case titles and kebab-case filenames.

**3. Code identifiers are never page titles.**
They are anchors: `[[backend/app/db/repositories/staff_users.py#StaffUserRepository]]`.
Sub-file pages would rot on every refactor with no stable key.

**4. `raw/` as immutable source material — REPLACED.**
`raw/` holds no copies. The raw corpus is the git working tree, addressed by repo-relative
path. It is immutable in the schema's sense (the wiki never edits it) and mutable in reality,
which is exactly why staleness is tracked by blob SHA rather than by log presence.

**5. Dedup by filename-presence in `log.md` — REPLACED.**
Stock dedup treats a *modified* file as already-ingested and skips it forever, and collides on
repeated basenames (`__init__.py`, `index.ts`). Here, outstanding work is always recomputed by
`brain-scan.sh` as `page.blob != git hash-object(file)`. `log.md` is never read to decide what
to ingest.

**6. "Search the wiki first, raw as a last resort" — OVERRIDDEN by the orientation/behavior
split** stated at the top of this file. The wiki answers *where and why*; the code answers
*what*. A page is trusted for orientation only while its `blob` matches.

**7. Flat 120-char index — REPLACED by a two-level generated index.**
`wiki/index.md` lists synthesis, concepts, and entities inline, but under `## Sources` lists
only directory pages, indented by depth. Leaf entries live in each directory's `_index.md`.

**8. The ingest skill's per-source confirmation gate — REPLACED by per-batch confirmation.**
The stock gate ("Wait for confirmation before proceeding") sits inside a per-source loop; at
350 sources that is 350 confirmations with no documented bulk mode. `/brain-ingest` keeps the
skill's output contract and replaces only its control flow.

**9. `kind: generated` pages are excluded from the default stale set.**
Lockfiles (`backend/uv.lock`, `frontend/pnpm-lock.yaml`) get stub pages so 1:1 coverage holds,
but re-summarizing machine output on every dependency bump is pure noise. `--all` includes them.

**10. `kind: vendored` + `applicability: vendored-inapplicable` are excluded from the default
stale set.** 199 of the 350 tracked files are Spartan/planetscale toolkit boilerplate, much of
it targeting stacks this repo does not use. They would all go stale together on the next
`spartan update`, turning one command into a 199-line drift report. `--all` includes them.

## This repo, in one paragraph

Boutique Platform — a Hebrew-first (RTL) multi-tenant SaaS for bridal and evening-wear
boutiques in Israel. Backend is Python 3.13 / FastAPI / SQLAlchemy 2 / Alembic / PostgreSQL
with forced per-tenant row-level security, managed by `uv`. Frontend is a pnpm 10 monorepo:
React 19 + Vite 8 + TypeScript 5.7 + Tailwind 4, apps `storefront` and `manage`, packages `ui`
and `api-client`. The root `Makefile` drives everything.

**Warning that every vendored page must carry:** `.claude/CLAUDE.md` and ~31 rule/agent/skill
files under `.claude/` describe a **Kotlin + Micronaut** backend and a **React + Next.js**
frontend. Neither exists here — `git ls-files | grep -E '\.(kt|kts|gradle)$'` returns zero and
`frontend/package.json` has no `next` dependency. Those files are unmodified Spartan toolkit
boilerplate written for other repos. See [[Documented Stack Vs Actual Stack]].
