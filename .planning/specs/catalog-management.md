# Spec: Feature 8 вАФ Catalog Management (Epic E2)

**Created**: 2026-07-24 ¬Ј **Revised**: 2026-07-24 (rev 2 вАФ post six-critic verification pass) ¬Ј **Status**: approved-for-build ¬Ј **Epic**: E2 Feature 8 ¬Ј **Effort**: M
**Depends on**: E1 #2 (media bucket provisioned; AWS account `not-started` вАФ see Risks), E1 #5 (owner auth), E2 #7 (manage console shell + `/manage` router, CSRF middleware, error handlers) вАФ branches off `main` ¬Ј **Feeds**: E2 #10 (storefront catalog grid + dress detail read this shape), E2 #9 (design restyle must be amended to cover the new Catalog tab вАФ see Risks 6), E5 #7 (date-bound reservation supersedes the manual `reserved` flag)

## Problem

F7 turned a provisioned boutique into a configured business, but the storefront has nothing to show. The E2 success criterion "customers browse the catalog and dress pages" and the storefront designs (`storefront-catalog.md`, `storefront-dress-detail.md`) were both drawn against a dress/variant/media shape that does not exist yet вАФ the catalog grid, the price-visibility treatment, the "„Ф„Х„Ц„Ю„Я" badge, the gallery, the size chips and the no-photo monogram state are all rendering data F8 must define. Nothing about photography can be faked: a bridal storefront without images is not a storefront, and image upload is the one part of this feature that touches infrastructure we do not own yet (the AWS account is `not-started`). The feature must therefore be buildable, mergeable and CI-verifiable **today**, with the production bucket reduced to configuration.

## Goal

The owner opens `{slug}вА¶/manage` вЖТ **„І„Ш„Ь„Х„Т**, creates a dress (name, description, price, price visibility, manual "„Ф„Х„Ц„Ю„Я" flag, catalog order), sets its size/quantity matrix from a standard EU bridal quick-entry list plus free-text custom sizes, uploads up to 12 photos straight from the browser to S3 via a presigned POST policy, orders the gallery, and archives (and un-archives) dresses. "Out of stock" is never typed by anyone вАФ it is derived from the variant rows. With no bucket configured the whole catalog stays usable: only the media write endpoints answer `503 MEDIA_NOT_CONFIGURED`.

## Design

### Data (migration `0006_catalog.py`, copying `0005_boutique_settings.py`'s structure: `_STANDARD` columns вАФ which already include `tenant_id` вАФ `_updated_at_trigger`, per-table `GRANT` + `enable_tenant_rls` loop)

`revision = "0006"`, `down_revision = "0005"`. `_STANDARD` and `_updated_at_trigger` are **re-declared locally** in `0006_catalog.py` вАФ migrations stay self-contained; only `enable_tenant_rls` is imported from `app.db.rls`, exactly as 0005 does. `downgrade()` drops `dress_media`, `dress_variants`, `dresses` in that order.

All three tables are tenant-scoped, so `test_every_tenant_id_table_has_forced_rls` picks up FORCE RLS automatically. No foreign keys (house rule вАФ `dress_id` integrity is app-level), TEXT not VARCHAR, TIMESTAMPTZ, soft delete, `uuid_generate_v4()` as the column default, money in integer agorot.

**CHECK-vs-constant rule (uniform, stated once).** Every numeric DB CHECK carries the lower bound plus an **absurdity ceiling fixed at exactly 10√Ч the service constant** вАФ INT4 headroom, so tightening product policy never needs a migration. `app/catalog/validation.py` carries the real cap. One deliberate exception: `dress_media.byte_size`, whose ceiling is 2√Ч because the value is also a security bound the POST policy enforces. Where the bound is a *security boundary* rather than a product bound (accepted content types, media status), the DB pins the exact set. Each constant in the constants table below names its DB counterpart, so the pair is reviewable in one place.

- **`dresses`** вАФ `name TEXT NOT NULL`, `description TEXT` (nullable), `price_agorot INTEGER CHECK (price_agorot IS NULL OR (price_agorot > 0 AND price_agorot <= 1000000000))` (nullable = "no price recorded"; 10√Ч `MAX_PRICE_AGOROT`; money integer agorot, never float), `price_visible BOOLEAN NOT NULL DEFAULT true` (the ¬І2 per-item visibility toggle; storefront renders "„Ю„Ч„Щ„® „С„™„Щ„Р„Х„Э" when false *or* when price is null), `reserved BOOLEAN NOT NULL DEFAULT false` (the manual, **date-less** owner flag вАФ Hebrew "„Ф„Х„Ц„Ю„Я"), `sort_order INTEGER NOT NULL DEFAULT 0`.
  Indexes: `idx_dresses_tenant_active ON dresses (tenant_id, sort_order, created_at DESC, id) WHERE deleted_at IS NULL` вАФ the four keys fully cover the documented list order `sort_order ASC, created_at DESC, id ASC`, so paging is stable under concurrent inserts; `idx_dresses_tenant_archived ON dresses (tenant_id, sort_order, created_at DESC, id) WHERE deleted_at IS NOT NULL` (the archive view + restore, same ordering). Updated_at trigger. **No unique index on `(tenant_id, name)`** вАФ boutiques legitimately carry the same designer model in two colours, and unlike `appointment_types` (which the booking UI picks by name) a dress is only ever addressed by id, so a duplicate name is a cosmetic annoyance, not a data-integrity hazard. The manage UI shows a non-blocking hint when a name already exists; the DB does not refuse the write.
- **`dress_variants`** вАФ `dress_id UUID NOT NULL`, `size_label TEXT NOT NULL`, `quantity INTEGER NOT NULL DEFAULT 0 CHECK (quantity >= 0 AND quantity <= 10000)` (10√Ч `MAX_VARIANT_QUANTITY`), `sort_order INTEGER NOT NULL DEFAULT 0`.
  Indexes: `idx_dress_variants_dress_active ON dress_variants (tenant_id, dress_id, sort_order) WHERE deleted_at IS NULL`; partial unique `idx_dress_variants_dress_size_unique ON dress_variants (tenant_id, dress_id, lower(size_label)) WHERE deleted_at IS NULL`. **`lower()` in the index, not the raw label**: the service normalises (strip, collapse internal whitespace) before writing, but that still lets `"US 6"` and `"us 6"` coexist as two stock buckets for one physical size. Hebrew has no case, so `lower()` is a no-op for the common path and a real guard for the Latin custom labels. Updated_at trigger.
- **`dress_media`** вАФ `dress_id UUID NOT NULL`, `storage_key TEXT NOT NULL`, `content_type TEXT NOT NULL CHECK (content_type IN ('image/jpeg','image/png','image/webp'))`, `byte_size INTEGER NOT NULL CHECK (byte_size > 0 AND byte_size <= 20971520)`, `status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','ready'))`, `sort_order INTEGER NOT NULL DEFAULT 0`.
  The content-type CHECK is deliberate DB enforcement, not duplication: the accepted set is a **security boundary** (an `image/svg+xml` object served from our bucket is stored XSS on the storefront), so widening it must be a deliberate migration, exactly as 0005 made the refund bounds structural. The 20 MiB `byte_size` ceiling is the 2√Ч absurdity guard; the real 10 MiB limit lives in `validation.py` and in the POST policy.
  `media_id` is **minted client-side in Python** (`media_id = uuid.uuid4()`) and passed as the explicit `id` on the INSERT, so `build_media_key(tenant_id, dress_id, media_id, content_type)` is computable *before* the row is written and `storage_key` is populated in the same, only statement. `uuid_generate_v4()` remains the column DEFAULT for any non-service writer. This follows the house precedent in `.memory/patterns/atomic-parent-child-across-rls.md`. There is no post-insert UPDATE of `storage_key`, and `storage_key` is never mutated after creation.
  Indexes: `idx_dress_media_dress_ready ON dress_media (tenant_id, dress_id, sort_order, id) WHERE deleted_at IS NULL AND status = 'ready'` (the gallery read path вАФ the only one F10 will touch); `idx_dress_media_pending ON dress_media (tenant_id, dress_id, created_at) WHERE deleted_at IS NULL AND status = 'pending'` (the per-dress stale-pending sweep inside the presign transaction вАФ tenant-first like every other index here, so one tenant's sweep never costs anything proportional to platform-wide upload traffic); plain unique `idx_dress_media_storage_key_unique ON dress_media (tenant_id, storage_key)` вАФ **a regression guard on the key builder, nothing more**: because the key embeds the row's own PK it can only collide if `build_media_key` is ever changed to drop `media_id`, and this index turns that into an `IntegrityError` in CI rather than two rows pointing at one object. It contributes nothing to confirm idempotency (confirm is an UPDATE, never an INSERT). Updated_at trigger.
- **Grants + RLS** вАФ the 0005 loop verbatim for all three tables: `GRANT SELECT, INSERT, UPDATE, DELETE ON {table} TO app_user` then `enable_tenant_rls(table)`. Nothing here is append-only, so no `REVOKE` dance (that pattern belongs to `terms_versions` / `platform_audit_log`).
- **ORM** вАФ `app/models/dress.py`, `app/models/dress_variant.py`, `app/models/dress_media.py` extending `StandardColumns, Base`; `DressMediaStatus` StrEnum (`PENDING = "pending"`, `READY = "ready"`) added to `app/models/constants.py`. Repositories `app/db/repositories/dresses.py`, `dress_variants.py`, `dress_media.py` follow `appointment_types.py` exactly (RLS + redundant explicit `tenant_id` predicate, `soft_delete` returning a bool, `refresh` after flush to pick up the trigger-maintained `updated_at`). `DressMediaRepository.insert_pending(session, *, tenant_id, dress_id, media_id, storage_key, content_type, byte_size)` takes `media_id` and `storage_key` as required arguments.
- **No stored dimensions** вАФ F8 does no image processing, so width/height would have to come from the client, and a client-reported dimension is a lie we would then render layouts against. The storefront's fixed 3:4 crop (`storefront-catalog.md`) is CSS; nothing needs the real pixels. Likewise no `alt` column: the passed design gate fixes alt text as "alt from the dress name" (`screens/design-system/README.md`).

### Derived status semantics

`out_of_stock` is **computed, never stored**: `total_quantity == 0` over the dress's non-deleted variant rows (equivalent to `not any(quantity > 0)` given the `quantity >= 0` CHECK) вАФ so **zero variants вЗТ `out_of_stock = true`**. `CatalogService._stock_summary` is the single formula; it is fed by a pre-aggregated row for lists and by loaded rows for the detail, never by a per-row query. It is serialised onto `DressResponse` alongside `total_quantity` and `variant_count`. Nothing is stored because the only writer of stock is the variant-matrix replace, which rewrites the entire set in one transaction вАФ a cached boolean would need either a trigger (invisible business logic in SQL) or a second write that can fail independently, and both create a state where the badge and the stock disagree. The variant rows are the single source of truth. **`out_of_stock` is a manage-only signal**: F10 does not consume it (see "Note for F10 (media + stock)").

`reserved` is stored, manual, boolean and **date-less**, and it is **independent of stock**: F8 exposes `out_of_stock: bool` + `reserved: bool` and **no single `status` string**, because the epic's three states are not mutually exclusive (a reserved dress can still have stock; an out-of-stock dress is usually not reserved) and collapsing them into one enum forces a precedence rule that neither storefront screen has. `storefront-catalog.md` shows "„Ф„Х„Ц„Ю„Я" as a badge and `storefront-dress-detail.md` keeps the booking CTA live for a reserved dress вАФ both are boolean reads, not an enum switch. `archived` (= `deleted_at IS NOT NULL`) is the third, orthogonal flag and is never returned to the storefront at all.

The DB CHECK is the absurdity ceiling above each, per the rule stated in Data.

### Storage port (`app/storage/`)

```python
@dataclasses.dataclass(frozen=True)
class PresignedPost:
    url: str
    fields: dict[str, str]

@dataclasses.dataclass(frozen=True)
class ObjectHead:
    content_type: str
    byte_size: int

class MediaStorage(Protocol):
    @property
    def is_configured(self) -> bool: ...
    def presigned_post(self, *, key: str, content_type: str,
                       exact_bytes: int, expires_in: int) -> PresignedPost: ...
    def signed_get_url(self, *, key: str, content_type: str,
                       filename: str, expires_in: int) -> str: ...
    async def head_object(self, *, key: str) -> ObjectHead | None: ...
    async def read_prefix(self, *, key: str, length: int) -> bytes: ...
    async def delete_object(self, *, key: str) -> None: ...
```

**Sync vs async, and why**: `presigned_post` and `signed_get_url` are pure local HMAC computation in botocore вАФ zero network I/O вАФ so they are plain `def`. Making them `async` would be a lie that costs an `await` per gallery item on every list response. `head_object`, `read_prefix` and `delete_object` are real network calls and boto3 is blocking, so they are `async def` on the Protocol and `S3MediaStorage` wraps the blocking client call in `anyio.to_thread.run_sync` (anyio already ships under Starlette вАФ no new dependency) so the event loop is never parked on S3.

**Failure contract вАФ the only place botocore exceptions exist.** Every `S3MediaStorage` network method wraps `botocore.exceptions.ClientError` / `BotoCoreError` and re-raises `MediaStorageUnavailableError`, carrying **no AWS-supplied text** (the original is logged with `tenant_id`/`dress_id`/`media_id`/`storage_key`, never returned). `head_object` returns `None` **only** for a genuine 404 / `NoSuchKey` head miss; a 500/503 from S3, an expired credential, a connect or read timeout is `MediaStorageUnavailableError`, never a silent "your upload never arrived". `NoCredentialsError` / `PartialCredentialsError` on any path map to `MediaNotConfiguredError` вАФ a bucket without usable credentials is operationally identical to no bucket and must degrade the same way rather than 500.

Implementations: **`S3MediaStorage`** (boto3 client pinned to `signature_version="s3v4"`, region + optional endpoint from settings) and **`UnconfiguredMediaStorage`** (`is_configured = False`; every method raises `MediaNotConfiguredError`). The unconfigured object is chosen over `None` so no call site grows a null check and no code path can forget one. Tests use a third implementation, `InMemoryMediaStorage`. **`S3MediaStorage.__init__` performs no network I/O and no credential resolution** вАФ it only builds the botocore client вАФ so `create_app()` stays safe to call in the fast suite regardless of local `.env` contents. The instance is built once in `create_app()` and hung on `app.state.media_storage`; `app/catalog/router.py` defines `def get_media_storage(request: Request) -> MediaStorage: return request.app.state.media_storage` (mirroring `get_boutique_service`), and fast tests substitute via `app.dependency_overrides[get_media_storage]`.

**Credentials are bearer material.** `PresignedPost.fields` (which carries `policy` + `x-amz-signature`, valid 300 s) and any signed GET URL (valid 900 s) are never logged, never included in an error message, and any future request/response-logging middleware must redact `/manage/dresses/*/media/*` bodies. Presign, dress-list and dress-detail responses are returned with `Cache-Control: no-store`, so a signed URL cannot land in a shared cache or a bfcache entry.

**Config** (new `Settings` fields; names recorded in `.env.example`): `MEDIA_BUCKET` (`str | None = None` вАФ presence is what `is_configured` means), `MEDIA_REGION` (`"il-central-1"`), `MEDIA_ENDPOINT_URL` (`str | None` вАФ MinIO in CI, and a seam for any S3-compatible provider), `MEDIA_FORCE_PATH_STYLE` (`False`; MinIO needs `True`), `MEDIA_PRESIGN_MAX_PER_WINDOW`, `MEDIA_PRESIGN_WINDOW_SECONDS`. **AWS credentials are deliberately NOT settings fields** вАФ `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / `AWS_SESSION_TOKEN` are read by boto3's own provider chain from the process environment (Railway env only, per F2's B3), so they never enter our config object, never appear in a `repr`, and never get logged.

A `model_validator` **fails startup** when (a) `MEDIA_BUCKET` is set with an empty `MEDIA_REGION`, (b) `MEDIA_ENDPOINT_URL` is set while `app_env == "production"` вАФ it is a CI/S3-compatible-provider seam and must be `None` against real AWS, (c) `MEDIA_ENDPOINT_URL` is set to a non-`https://` URL outside `dev`, or (d) `MEDIA_FORCE_PATH_STYLE` is `True` while `app_env == "production"`. **A missing bucket is never a boot failure** (binding decision #5) вАФ but a *wrong* endpoint is, mirroring `_require_real_base_domain_outside_dev`.

Because `Settings.model_config` is `extra="ignore"`, a typo'd `MEDIA_BUKCET` is discarded silently. So the graceful degradation must be **observable**: `create_app()` logs one INFO line at wiring time вАФ `media storage configured: bucket=%s region=%s endpoint=%s`, or `media storage NOT configured вАФ upload endpoints will answer 503 MEDIA_NOT_CONFIGURED` вАФ and `/health` gains `"media": "configured" | "unconfigured"` (never the bucket name), which is what F2's staging smoke checks before attempting the presignвЖТuploadвЖТconfirmвЖТsigned-GET run.

**Key scheme**: `tenants/{tenant_id}/dresses/{dress_id}/media/{media_id}{ext}`. Tenant prefix first satisfies the checklist's "S3 keys tenant-prefixed" and leaves room for a future IAM `s3:prefix` condition per tenant. `{ext}` comes from `ACCEPTED_CONTENT_TYPES`. The presign request body is exactly `{content_type, byte_size}` вАФ there is **no field the client can send that influences the key**; `tenant_id` comes only from `get_current_tenant(request)` (host-derived) and `dress_id` only from a dress row this tenant actually owns (see Authorization invariants). `fields["key"]` necessarily exposes `tenant_id` to the browser: acceptable, because a tenant id is an identifier and not a secret, and no F8 endpoint accepts a tenant id from the client. The original filename is not stored at all in v1.

**Media origin isolation (standing invariant).** Media is served from an origin that is **never the tenant cookie domain nor any subdomain of it**. If a CDN or custom domain is ever placed in front of the bucket it must use a **separate registrable domain** вАФ `media.<domain>` is same-*site* to the `{slug}.<domain>` session cookie, which is precisely the case `app/csrf.py`'s header comment documents as dangerous. This constraint is inherited by the deferred CDN work in Risk 3 and must be restated in that feature's spec.

**Not configured**: `presign`, `confirm` and `media delete` return `503 MEDIA_NOT_CONFIGURED` with the fixed body `{"error":{"code":"MEDIA_NOT_CONFIGURED","message":"Image uploads are not available."}}`. Everything else keeps working вАФ dress CRUD, variants, reorder (a pure DB operation), and reads, where `MediaResponse.url` serialises as `null`. `DressDetailResponse.media_uploads_enabled` tells the UI up front (folded into the detail response rather than a separate `GET /manage/catalog/capabilities` вАФ one fewer round-trip, one fewer endpoint to secure, and the UI still handles a 503 defensively in case config changes mid-session). No bucket, region, endpoint, env-var or IAM identifier appears in any user-facing message.

### Presigned POST policy

`generate_presigned_post` (POST policy, **not** PUT вАФ binding decision #4: only a POST policy makes S3 itself enforce exact key **and** exact Content-Type **and** a length range). Conditions carried:

| Condition | Value | What it stops |
|---|---|---|
| `["eq", "$key", <server key>]` | the exact computed key | a client writing outside its tenant prefix, or over another dress's object |
| `["eq", "$Content-Type", <declared type>]` | one of the three accepted types | uploading an SVG/HTML payload under a jpeg presign |
| `["content-length-range", byte_size, byte_size]` | **the exact declared size** | zero-byte probes, bucket-filling, **and any object whose size differs from the row that authorised it** |
| `ExpiresIn` | `PRESIGN_TTL_SECONDS` = 300 | a leaked policy staying useful |

The length range is **exact, not `MINвА¶MAX`**. `MIN_UPLOAD_BYTES` / `MAX_UPLOAD_BYTES` remain the `validation.py` bounds on the *declared* value; pinning the policy to that declared value is what makes "declare 1 KiB, post 10 MiB" вАФ a deliberate way to plant a large orphan the DB can never find again вАФ structurally impossible. It also makes `MEDIA_MISMATCH`-on-size unreachable, leaving that code for content-type and magic-byte mismatches only.

**No `x-amz-server-side-encryption` condition or field.** Encryption is a **bucket property, not a form field the browser must remember to send**: S3 has applied SSE-S3 to every new object by default since January 2023, and pinning the header would additionally break MinIO (which rejects `x-amz-server-side-encryption` without a configured KMS) and every other S3-compatible target. **F2 ¬ІB3 requirement instead**: the bucket sets default encryption (SSE-S3) and a bucket policy `Deny s3:PutObject` when `s3:x-amz-server-side-encryption` is absent вАФ see Risks 2.

**No `acl` field, deliberately**: the bucket is Block-Public-Access + Object Ownership "bucket owner enforced", where ACLs are disabled and sending an `acl` form field makes S3 reject the POST with `400 InvalidRequest`. Anyone tempted to "harden" this by adding `["eq", "$acl", "private"]` will break every upload.

Returned to the browser: `url` plus `fields` = `key`, `Content-Type`, `policy`, `x-amz-algorithm`, `x-amz-credential`, `x-amz-date`, `x-amz-signature`. The client posts `multipart/form-data` with **all fields first and `file` last** (S3 ignores anything after `file`).

**Read-side safety.** `signed_get_url` sets `ResponseContentType` to the stored `content_type` **and `ResponseContentDisposition=attachment; filename="{media_id}{ext}"`, not `inline`**. `Content-Disposition` is ignored for subresource loads, so `<img src>` still renders in the manage gallery and in F10's storefront; `attachment` only turns a *top-level navigation* to a signed URL from render into download вАФ exactly the polyglot vector. Three layers guard content: the closed content-type allowlist pinned by the POST policy, this response override plus media-origin isolation, and the magic-byte assertion at confirm (below). S3 cannot emit `X-Content-Type-Options: nosniff` on a presigned GET (the override set is limited to response-content-type/-disposition/-encoding/-language/-cache-control/-expires), so these three are the whole defense and none of them is optional.

### API (`app/catalog/` module: `router.py`, `schemas.py`, `service.py`, `validation.py`, `keys.py`)

`APIRouter(prefix="/manage")`, registered in `create_app()` after the boutique router. Every route takes `Staff = Depends(get_current_staff)` + `get_current_tenant(request)`; every tenant-table access runs inside `tenant_session(session_factory, tenant.id)`. `CatalogService` is constructed in `create_app()` (session factory, `media_storage`, a `FixedWindowRateLimiter` built from the two presign settings with `clock=time.monotonic`, and `pending_ttl_seconds`), hung on `app.state.catalog_service`, and read by a `get_catalog_service(request)` dependency exactly as `get_boutique_service` does. FastAPI-idiomatic verbs, matching F7 (the Micronaut "no path params / no REST verbs" rules in `.claude/rules/` do not apply to this codebase).

**Authorization invariants (read before the table).** Every `/manage/dresses/{dress_id}/...` route **resolves the parent dress first**: the first statement inside `tenant_session` вАФ and, for presign, inside the advisory lock вАФ is `dress = await dresses.by_id(session, tenant.id, dress_id)`; `if dress is None: raise CatalogNotFoundError`. `by_id` carries `tenant_id == tenant.id`, `id == dress_id` **and** `deleted_at IS NULL` (the `appointment_types.by_id` predicate verbatim), so an unknown id, an archived dress and another tenant's dress are one indistinguishable 404. `build_media_key(...)` is called only after that check returns a row, so no unverified id can reach an S3 key. Every media repository method takes `(session, tenant_id, dress_id, media_id)` and puts all three in the `WHERE` alongside `deleted_at IS NULL`: a media id that exists but belongs to a *different dress of the same tenant* is `404 NOT_FOUND`, never a silent re-parent. The reorder permutation is computed over `(tenant_id, dress_id, status='ready', deleted_at IS NULL)`.

**Transaction boundaries (invariant, stated once).** **No `MediaStorage` network method (`head_object`, `read_prefix`, `delete_object`) may ever be awaited inside a `tenant_session` block.** `tenant_session` is `async with session_factory() as session, session.begin()` вАФ the whole block is one Postgres transaction, and boto3's default 60 s connect + 60 s read timeouts would pin a pool connection (and a per-dress advisory lock) across an S3 stall. `presigned_post` and `signed_get_url` may run anywhere: they are local HMAC. Mutations that return `DressDetailResponse` do their write transaction, then a short read transaction to build the view; signed URLs are minted after the session closes.

**Per-dress advisory locks.** Every write path touching `dress_media` for a dress вАФ presign, confirm-promote, media delete, media reorder вАФ takes `pg_advisory_xact_lock(hashtext('dress-media:' || :dress_id))` in its write transaction. The variants replace takes `pg_advisory_xact_lock(hashtext('dress-variants:' || :dress_id))`. Two distinct prefixes, so photo work and stock work on the same dress never serialise against each other, and both stay clear of F7's un-prefixed `hashtext(:tenant_id)` lock space.

**List query shape.** A page of `GET /manage/dresses` is exactly four statements: (1) the dress page `WHERE tenant_id = :t AND deleted_at IS [NOT] NULL [AND name ILIKE :q] ORDER BY sort_order, created_at DESC, id LIMIT/OFFSET`; (2) its `COUNT(*)`; (3) `SELECT dress_id, count(*) AS variant_count, coalesce(sum(quantity),0) AS total_quantity FROM dress_variants WHERE tenant_id = :t AND dress_id = ANY(:ids) AND deleted_at IS NULL GROUP BY dress_id`; (4) `SELECT DISTINCT ON (dress_id) *, count(*) OVER (PARTITION BY dress_id) AS media_count FROM dress_media WHERE tenant_id = :t AND dress_id = ANY(:ids) AND deleted_at IS NULL AND status = 'ready' ORDER BY dress_id, sort_order, id` for `cover` + `media_count`. Statements 3 and 4 are served by the leading columns of `idx_dress_variants_dress_active` and `idx_dress_media_dress_ready`. A dress with no matching group row gets `out_of_stock = true, total_quantity = 0, variant_count = 0, media_count = 0, cover = null` (edge case 6).

**Presign sequence.** (1) Throttle: `key = f"presign:{tenant_id}"`; `if limiter.is_blocked(key): raise MediaPresignThrottledError` вЖТ 429, before any work. (2) `tenant_session` #1, media advisory lock: resolve the dress (404); `SELECT` stale pendings (`status='pending' AND created_at < now() - pending_ttl`) capturing their `storage_key`s and soft-delete them; count ready + young-pending; `>= MAX_MEDIA_PER_DRESS` вЗТ `MediaLimitReachedError`; mint `media_id`, build the key, `insert_pending`. Commit. (3) **Outside any session**: best-effort `delete_object` for each swept key (the sweep is the only holder of those keys, so this is what bounds the orphan window to `PENDING_MEDIA_TTL_SECONDS` + the next presign for that dress rather than forever); then `presigned_post(...)`; then `limiter.record_failure(key)` вАФ `FixedWindowRateLimiter` counts **only what is explicitly recorded** (its docstring: "successes never count"), so a successful presign, which authorises a 10 MiB write to our bucket, must be recorded by hand. This is the `create_terms_version` precedent (`app/boutique/service.py:325`).

**Confirm sequence.** (1) `tenant_session` #1: resolve dress and media row by `(tenant_id, dress_id, media_id)`; if already `ready`, build and return the detail view and stop вАФ the idempotent short-circuit. (2) **Outside any session**: `head = await storage.head_object(key=row.storage_key)`; `None` вЗТ `MediaNotUploadedError`; `head.content_type != row.content_type` вЗТ best-effort `delete_object` then `MediaMismatchError`; `await storage.read_prefix(key=вА¶, length=16)` must match `MAGIC_PREFIXES[row.content_type]`, else best-effort `delete_object` then `MediaMismatchError`. (An object that fails verification must not persist вАФ that is the second half of the orphan story.) (3) `tenant_session` #2, media advisory lock: re-read the row and, **if still `pending`**, set `status='ready'` and `sort_order = coalesce(max(sort_order), -1) + 1` over this dress's ready rows; otherwise leave it. Build the detail view from the same transaction's rows. Confirm is idempotent **by construction** вАФ it is an UPDATE of an existing row guarded by a `status = 'pending'` predicate plus the re-read inside the locked promote transaction. The `storage_key` unique index has nothing to do with it.

**Every mutation that changes a derived field returns the full detail view**, so the console never needs a follow-up GET. `DELETE /manage/dresses/{dress_id}` correctly stays `OkResponse` вАФ the dress is gone.

**Note for F10 (media + stock).** `VariantResponse.quantity` is a **manage-only** field. **UI behaviour.**

*State ownership.* `CatalogSection` owns the list; `DressEditor` owns the detail. Every mutation response is a `DressDetail`; the editor replaces `detail` from it and calls `onDressChanged(detail)`, which patches the matching list row in place (`setDresses(dresses.map(d => d.id === next.id ? next : d))`) вАФ F7's `TypesSection` pattern. No list refetch, and the derived `„Р„Ц„Ь „Ю„Ф„Ю„Ь„Р„Щ` / `„Ф„Х„Ц„Ю„Я` badges and cover thumbnail can never disagree between the two views.

*Stock badge is three-way*, derived client-side, read-only in both views: `variant_count == 0` вЖТ neutral "„Ь„Р „Ф„Х„Т„У„®„Х „Ю„Щ„У„Х„™"; `variant_count > 0 && total_quantity == 0` вЖТ warning "„Р„Ц„Ь „Ю„Ф„Ю„Ь„Р„Щ"; otherwise "„С„Ю„Ь„Р„Щ (N)". Without the first case, a boutique entering 60 dresses before filling any size matrix sees a full page of "„Р„Ц„Ь „Ю„Ф„Ю„Ь„Р„Щ" on brand-new dresses. No API change вАФ `variant_count` and `total_quantity` already carry it.

*Upload queue.* Each queued file renders a row with its file name, size (`<bdi>` LTR) and one of five states: `„Ю„Ю„™„Щ„Я` ¬Ј `„Ю„Ґ„Ь„ФвА¶` ¬Ј `„Ю„Р„Ю„™вА¶` (the confirm call) ¬Ј `„Ф„Х„Ґ„Ь„™„Ф` ¬Ј `„†„Ы„©„Ь„Ф вАФ {reason}` + `„†„°„Щ „©„Х„С`. **A byte-level progress percentage is explicitly out of scope**: `fetch` exposes no upload-progress event and F8 adds no XHR-based uploader or upload library, so `„Ю„Ґ„Ь„ФвА¶` is an indeterminate indicator. Because uploads are sequential, exactly one row is ever non-terminal, and the queue plus a `role="status"` summary (`„Ю„Ґ„Ь„Ф 3 „Ю„™„Х„Ъ 8`) is the progress affordance.

*Failure recovery.* On any failure вАФ S3 non-2xx, network/CORS rejection, or a confirm error other than a transient 5xx вАФ `MediaGallery` calls `api.deleteMedia(dressId, mediaId)` to **release the pending row** (which is why media DELETE accepts `pending` rows), then renders a Hebrew reason plus a `„†„°„Щ „©„Х„С` button that re-runs the full presign вЖТ POST вЖТ confirm cycle for that file (a retry always takes a fresh presign вАФ the previous policy is dead once its 5-minute TTL elapses). Before opening the picker the gallery blocks selecting more than `media_slots_remaining` files inline (`„†„Щ„™„Я „Ь„Ф„Ґ„Ь„Х„™ „Ґ„У 12 „™„Ю„Х„†„Х„™ „Ь„©„Ю„Ь„Ф вАФ „†„Х„™„®„Х 3`), so `MEDIA_LIMIT_REACHED` is only ever a race; when it does arrive the message is "„Ф„Т„Ь„®„Щ„Ф „Ю„Ь„Р„Ф. „Щ„Щ„™„Ы„Я „©„Ф„Ґ„Ь„Р„Х„™ „©„†„Ы„©„Ь„Х „Ґ„У„Щ„Щ„Я „™„Х„§„°„Х„™ „Ю„І„Х„Э вАФ „®„Ґ„†„†„Щ „Р„™ „Ф„Ґ„Ю„Х„У „Х„†„°„Щ „©„Х„С." Without this path a few failed uploads on a flaky mobile uplink lock the gallery for an hour with no button that fixes it.

*Reorder accessibility.* вЖС/вЖУ buttons, **no drag-and-drop in v1** (IS 5568 is a legal floor and drag is the least accessible reorder affordance). Each item's buttons carry an explicit `aria-label` (`„Ф„Ц„Ц„™ „™„Ю„Х„†„Ф 2 „Р„Ч„Х„®„Ф` / `„Ф„Ц„Ц„™ „™„Ю„Х„†„Ф 2 „І„У„Щ„Ю„Ф`); the first item's вЖС and the last item's вЖУ are `disabled`; after the optimistic swap the component **restores focus to the same logical button** via a ref keyed by `media.id` вАФ never let focus drop to `<body>`, which is what makes naive button reordering worse than drag. A single `role="status"` region under the gallery announces the result (`„Ф„™„Ю„Х„†„Ф „Ф„Х„Ґ„С„®„Ф „Ь„Ю„І„Х„Э 3 „Ю„™„Х„Ъ 5`) and also carries the upload-queue states. Reorder is optimistic with rollback on error, then a refetch on `MEDIA_ORDER_MISMATCH`.

*Signed-URL expiry.* Gallery `<img>` URLs expire after 15 minutes. A single `hasRefreshedRef` guard per gallery mount means the first `onError` from any image triggers exactly one `GET /manage/dresses/{id}` and subsequent errors are ignored; if an image still fails after that refresh it renders the no-photo placeholder rather than retrying. Gallery images take `alt={\`${dress.name} вАФ „™„Ю„Х„†„Ф ${index + 1}\`}`.

*Storage-disabled state.* When `media_uploads_enabled` is false (or any media call returns 503), the gallery renders a `role="status"` bordered notice вАФ "„Ф„Ґ„Ь„Р„™ „™„Ю„Х„†„Х„™ „Ґ„У„Щ„Щ„Я „Ь„Р „Ц„Ю„Щ„†„Ф" вАФ with the file input `disabled` **and** the same explanation on its `<label>` (`disabled` alone removes the control from the tab order, so a screen-reader user would never learn why). The rest of the editor stays fully interactive. Nothing in the catalog tab becomes unusable because a bucket is missing.

## Edge cases

1. **Concurrent variant replaces** on the same dress вЖТ serialised by `pg_advisory_xact_lock(hashtext('dress-variants:' || dress_id))`; validation failures leave the previous matrix untouched. Two different dresses never contend, and a photo operation on the same dress never contends either (distinct lock prefix).
2. **Concurrent presigns racing the cap** вЖТ both run inside the same per-dress media advisory lock, so the 13th presign loses and gets `MEDIA_LIMIT_REACHED`, never an over-full gallery.
3. **Abandoned upload**: the presign succeeded, the browser died. The `pending` row stops counting toward the cap after 1 h and is swept on the next presign for that dress вАФ and the sweep, which holds the row's `storage_key`, best-effort deletes the object too, so the orphan window is bounded rather than permanent. The UI's own failure path deletes the pending row immediately, so this is the abandoned-tab case, not the failed-upload case.
4. **Confirm without an upload** вЖТ `head_object` returns `None` вЖТ `409 MEDIA_NOT_UPLOADED`, row stays `pending` and expires normally. An S3 *outage* is a different answer: `503 MEDIA_STORAGE_UNAVAILABLE`, row untouched and still confirmable.
5. **Confirm retried after a lost response** вЖТ idempotent 200 with the same detail view: confirm is an UPDATE guarded by `status = 'pending'` plus a re-read inside the locked promote transaction, so a retry cannot double-bump `sort_order` and cannot fork a row.
6. **Dress with zero variants** вЖТ `out_of_stock = true`, `total_quantity = 0`, `variant_count = 0`. Correct per the derivation rule, and what a freshly created dress looks like вАФ the manage UI shows "„Ь„Р „Ф„Х„Т„У„®„Х „Ю„Щ„У„Х„™", not "„Р„Ц„Ь „Ю„Ф„Ю„Ь„Р„Щ".
7. **Archive then restore** вЖТ the archive UPDATEs carry `WHERE deleted_at IS NULL`, so a photo the owner deleted individually keeps its older stamp; restore reads the dress's `deleted_at` first and matches children on it. Photos and sizes deleted individually before the archive stay deleted. A restored dress rejoins the list with its ordering intact.
8. **Duplicate size by case/whitespace** (`" 38"`, `"US 6"` vs `"us 6"`) вЖТ normalised, then rejected by the `lower(size_label)` partial unique index as `409 DUPLICATE_SIZE`.
9. **Cross-tenant probes**: tenant A requesting B's `dress_id` or `media_id` вАФ read, update, archive, restore, presign, confirm, delete, reorder вАФ all return `404 NOT_FOUND`, indistinguishable from a missing row. **Within one tenant**, dress A's media id addressed through dress B's URL is also 404. Every new repository method and endpoint gets an entry in the permanent isolation suite.
10. **Bucket not configured** вЖТ dress and variant CRUD fully usable, reads return `url: null`, only the media write endpoints answer `503 MEDIA_NOT_CONFIGURED`. The app boots normally and logs the fact (binding decision #5).
11. **Tampered upload**: the client edits the `key` or `Content-Type` form field, or posts a file of a different size than it declared вЖТ **S3** rejects it (403 / `EntityTooLarge` / `EntityTooSmall`), not us. Our DB row stays `pending` and the UI's failure path deletes it. This is the whole point of the POST policy.
12. **Content-type-honest polyglot**: an HTML/JS payload declared and posted as `image/jpeg` at exactly the declared size passes the policy вЖТ confirm's 16-byte magic-prefix check fails вЖТ the object is deleted and `409 MEDIA_MISMATCH` is returned. Even if it were stored, `attachment` disposition plus media-origin isolation keep it non-executable.
13. **Signed URL expiry during a long editing session** вЖТ the `<img>` `onError` path refetches the dress exactly once and re-renders with fresh URLs.
14. **Price null with `price_visible = true`** вЖТ valid; the storefront shows "„Ю„Ч„Щ„® „С„™„Щ„Р„Х„Э" (visibility is a permission, not a promise that a price exists). The editor's preview line makes this visible at write time.
15. **Repeat DELETE / repeat restore** вЖТ `404 NOT_FOUND` (the UPDATEs are predicated on `deleted_at IS NULL` / `IS NOT NULL`), matching F7's `archive_appointment_type`. The console disables the button while in flight and treats a 404 from archive/restore/delete as "already done" rather than surfacing an error toast.
16. **Bucket set, credentials missing** (the exact state F8 ships into) вЖТ `NoCredentialsError` maps to `503 MEDIA_NOT_CONFIGURED`, not a 500.

A **session-scoped** `minio_container` fixture in `tests/conftest.py`, guarded by the same `_docker_running()` check that guards `postgres_url`, starts **`minio/minio:RELEASE.2025-04-22T22-12-26Z`** (pinned like `postgres:16-alpine` вАФ bumping it is a deliberate commit), creates `boutique-test-media` **using the boto3 client we already depend on** (not the `minio` SDK вАФ no second untyped import in `tests/`), and yields endpoint + credentials only, never a module-level client, so collection with `-m "not db"` never starts it. `S3MediaStorage` is pointed at it via `MEDIA_ENDPOINT_URL` + `MEDIA_FORCE_PATH_STYLE=true`. Budget: roughly 45вАУ75 s added to the backend CI job (cold image pull dominates); if it exceeds ~2 min, split `s3` into its own CI step rather than loosening the assertions.

Assertions, with a real multipart POST from `httpx`:

| Assertion | Expected |
|---|---|
| happy path: presign вЖТ POST (fields then `file`) вЖТ `head_object` вЖТ `read_prefix` вЖТ confirm | 204 upload, object present, row `ready` |
| declared `image/jpeg`, posted `Content-Type: image/png` | **403** вАФ policy rejects |
| declared 1 KiB, posted 1 MiB | **EntityTooLarge** вАФ the exact `content-length-range` rejects |
| posted smaller than declared | **EntityTooSmall** вАФ policy rejects |
| client rewrites the `key` field (own tenant prefix вЖТ another tenant's) | **403** вАФ policy rejects |
| POST after expiry вАФ signed directly via `S3MediaStorage` with `expires_in=1` + `time.sleep(1.5)` | **403** вАФ policy rejects |
| signed GET fetches the object; the same URL after its TTL (`expires_in=1` + sleep) | 200 with `Content-Disposition: attachment`, then 403 |
| honest-size HTML payload declared `image/jpeg` вЖТ confirm | `MEDIA_MISMATCH`, object deleted |
| `delete_object` then `head_object` | `None` |

The two expiry rows use `expires_in=1` deliberately: they test S3's *enforcement* of the expiry field, not our default value вАФ which is covered without network by the recording-fake assertions in `test_storage_port.py`. Sleeping 300 s and 900 s in CI is not an option.

**Tooling parity**: the `Makefile` `lint` target's backend line becomes `uv run ruff check . && uv run ruff format --check . && uv run mypy app tests`, matching CI вАФ F8 adds five substantial new test modules using boto3, httpx multipart and Testcontainers, the surface most likely to have type errors, and today `make lint` typechecks only `app`.

## Risks & open items

1. **AWS account is `not-started`** (`external-applications.md` row 1 вАФ the standing re-flag rule applies, and this spec re-flags it). F8 merges and is fully CI-green against MinIO; the staging/production bucket is then pure configuration. **The binding gate is F10, not F8.** F8 merges without the account; F10 cannot ship meaningfully without it, since every approved storefront screen is photography-led ("the grid IS the hero"). **Gate**: the AWS account must be `done` and the staging presignвЖТuploadвЖТconfirmвЖТsigned-GET smoke green **before F10's spec is written**, otherwise E2 closes on a monogram-only storefront that technically satisfies criterion 3 while being unshippable. Pilot-onboarding consequence: do not have the boutique enter catalog data before the bucket is live вАФ dress and variant CRUD work, but the owner would have to revisit all ~60 dresses to add photos. Owner: user (only the user can file it). Trigger: now.
2. **F2 ¬ІB3 CORS вАФ amended (done, 2026-07-24)**; `AllowedMethods` already reads `[POST, GET, HEAD]` with the F8 amendment note inline. **No further spec edit is needed** вАФ a reviewer re-doing it risks a conflicting second edit. Two items remain: (a) ¬ІB3 must additionally record the **bucket-level SSE requirement** replacing F8's dropped form field вАФ default encryption (SSE-S3) plus a bucket policy `Deny s3:PutObject` when `s3:x-amz-server-side-encryption` is absent; (b) verification вАФ the first staging smoke after the bucket lands must run presign вЖТ browser-shaped POST вЖТ confirm вЖТ signed GET from a `https://*.<staging-domain>` origin and confirm the preflight passes. Until then every CORS assertion in this feature is server-side only: MinIO exercises the policy, not the browser preflight. Owner: F2 follow-up. Trigger: AWS account `done`.
3. **No image processing** вАФ a 10 MiB original is served both to a bride on cellular *and* to the owner's own phone in the manage list, which is the heavier path in v1 (24 covers √Ч up to 10 MiB). Mitigations in scope: the 10 MiB cap, `DRESS_LIST_DEFAULT_LIMIT = 24` with `loading="lazy"`, `SIGNED_GET_TTL_SECONDS` letting the browser cache within a session, **and the in-product upload guidance line in `MediaGallery`** вАФ the epic's stated mitigation, now a shipped artifact rather than a document. A thumbnail/CDN feature is a known follow-up, bound by the media-origin-isolation invariant, and should be scheduled before onboarding boutique #2, not before the pilot.
4. **Orphan S3 objects.** A noncurrent-version lifecycle rule does **not** reach these вАФ an abandoned or failed-verification object is the *current* version of its key, and a current-version expiry rule on the media prefix would delete live gallery images, so it is not a safe substitute. F8 therefore reduces orphans structurally rather than reaping them: the stale-pending sweep deletes the objects it is about to orphan, confirm deletes the object it rejects, the UI's failure path deletes the pending row (and hence its object) immediately, and the exact `content-length-range` removes the deliberate large-orphan path. What remains is a failed `delete_object` after a committed soft-delete, logged at WARNING with `tenant_id`/`dress_id`/`media_id`/`storage_key` вАФ that log line is the future reconcile job's input, so swallowing it silently is not acceptable. Owner: E4 #21 infra runbook. Trigger: measurable orphan volume, or the first bucket cost review.
5. **`archived` growth** вАФ soft-deleted dresses accumulate forever, as everywhere else in this codebase. The partial indexes keep the active paths clean; PPL erasure is a platform-wide job (checklist item), not F8's.
6. **F9 `manage-restyle.md` amendment required** вАФ F8 adds a fifth console section "„І„Ш„Ь„Х„Т" to a design that passed its gate against four (`profile | hours | types | terms`), and `manage-restyle.md`'s per-component table has no rows for `CatalogSection`, `DressEditor`, `VariantMatrix` or `MediaGallery`, nor a console-wide upload-queue or storage-disabled state. **Ruling (decided, not deferred): Catalog does NOT join `SetupProgress`** вАФ setup means the configuration prerequisites for accepting bookings, catalog is ongoing content, and `storefront-catalog.md` already ships a dignified zero-dress state ("„Ф„І„Х„Ь„І„¶„Щ„Ф „С„У„®„Ъ"), so the checklist stays 4-item and its denominators are unchanged. ## Decisions Log

**HIGH findings вАФ all applied.**
1. *Unbuildable `storage_key`* вАФ `media_id` is minted client-side (`uuid.uuid4()`) and passed as the explicit `id`, so the key exists before the single INSERT; `insert_pending` takes both as required args.
2. *Client-controlled `dress_id` in the S3 key* вАФ added the **Authorization invariants** paragraph: the parent dress is resolved (with `deleted_at IS NULL`) before `build_media_key` is ever called; media lookups carry `(tenant_id, dress_id, media_id)`.
3. *Inert presign throttle* вАФ `FixedWindowRateLimiter` records only what it is told, so `record_failure` is called after **every successful** presign (`create_terms_version` precedent); limiter wired in `create_app()`; 429 test added.
4. *SSE condition breaks MinIO and buys nothing* вАФ dropped the `x-amz-server-side-encryption` condition and field; encryption becomes an F2 ¬ІB3 bucket property + `Deny` policy. Policy-shape test now asserts **no** SSE condition.
5. *No error path for "bucket configured but S3 failed"* вАФ `MediaStorageUnavailableError` вЖТ 503 `MEDIA_STORAGE_UNAVAILABLE` with a fixed body and no AWS text; `head_object` returns `None` only on a genuine 404; `NoCredentialsError` вЖТ 503 `MEDIA_NOT_CONFIGURED`.
6. *Overstated sniffing defense* вАФ `signed_get_url` now uses `attachment` disposition; added the standing **media-origin-isolation** invariant; added a 16-byte magic-prefix check at confirm (`read_prefix` on the port) so the allowlist is real rather than declared.
7. *Handler resolution by class identity* вАФ new `app/errors.py` base classes, boutique errors re-parented, handlers re-registered on the bases, and every new class listed with its status; a test asserts every code's status and shape.
8. *S3 calls inside `tenant_session`* вАФ stated the **transaction-boundaries** invariant and spelled presign, confirm and delete as explicit multi-step sequences with all network I/O outside the session.
9. *False `storage_key`-index guarantee hiding a real race* вАФ index restated as a key-builder regression guard; idempotency re-justified on the `status='pending'` predicate + locked re-read; the per-dress media advisory lock extended to confirm, delete and reorder with a distinct prefix from the variants lock.
10. *List N+1* вАФ pinned the four-statement list query shape and added a statement-count regression test.
11. *Partial mutation responses leave derived badges stale* вАФ variants replace, confirm, media delete and reorder all return `DressDetailResponse`; state ownership and `onDressChanged` patching specified.
12. *Upload cap trap* вАФ media DELETE accepts `pending` rows as the client's abort path; per-file retry, client-side `media_slots_remaining` precheck, and a specific Hebrew message on the race.
13. *`s3` marker breaks the no-Docker loop* вАФ `s3` is always applied alongside `db`; both registered in `pyproject.toml`; `DOCKER_HELP` updated.
14. *boto3 breaks mypy* вАФ added the `boto3.*`/`botocore.*` override.
15. *Bounds duplicated in Settings, DB and validation.py* вАФ deleted `MEDIA_MAX_UPLOAD_BYTES` / `MEDIA_PRESIGN_TTL_SECONDS` / `MEDIA_SIGNED_GET_TTL_SECONDS` from `Settings`; product policy lives once in `validation.py`, mirrored to `validation.ts` and guarded by a parity test.
16. *Expiry tests would sleep 20 min* вАФ MinIO expiry rows sign with `expires_in=1`; the production defaults are asserted separately against a recording fake.
17. *Confirm idempotency only tested against a fake* вАФ moved to `test_catalog_integration.py` with row-count and `sort_order` assertions plus a concurrent case.
18. *`head_object` swallowing outages* вАФ contract pinned: `None` only for a 404 head miss.

**Adjudicated conflicts.**
- *`include_archived` semantics* вАФ DB critic (exclusive `archived: bool`, each branch on one partial index) beats API critic (combined active+archived view): a combined branch can use neither partial index, and both the frontend critic and the product critic describe the UI as a *toggle*. Renamed to `archived`; API critic's visibility rule (detail + restore resolve archived, everything else 404) applied on top.
- *`media_count` semantics* вАФ three critics proposed `media_count`=ready (DB), `+pending_media_count` (DB/frontend) and `+media_slots_remaining` (API). Adopted ready-only `media_count` **plus** `media_slots_remaining`; dropped `pending_media_count` as strictly redundant with the number the client actually needs.
- *Orphan reconcile* вАФ security critic's DB-driven deletes (sweep + confirm-mismatch) are in scope; testability critic's `media-reconcile` CLI is deferred to Risk 4 with owner E4 #21, because it needs list-bucket IAM that does not exist yet and would be untestable in F8.
- *Storage-error code name* вАФ `MEDIA_STORAGE_UNAVAILABLE` (testability critic) over `MEDIA_STORAGE_ERROR` (security critic): 503 means unavailable, and the code should say so.
- *`sort_order` on dresses* вАФ three critics offered expose-or-drop. Exposed (option a): the column and index already exist, the storefront grid reads the ordering, and a pinned signature gown is worth one numeric input. `UpdateDressRequest` carries it; `DressEditor` gains the field; the index extended to `(tenant_id, sort_order, created_at DESC, id)`.
- *`MEDIA_ORDER_MISMATCH` status* вАФ 400 вЖТ **409**: the payload is well-formed and conflicts with state, matching every other media conflict and the FE's already-specified refetch behaviour.
- *`DRESS_LIST_DEFAULT_LIMIT`* вАФ 50 вЖТ **24** (frontend critic's egress argument); `DRESS_LIST_MAX_LIMIT` stays 100 with the router signature spelled out so F7's default==max shortcut is not copied.
