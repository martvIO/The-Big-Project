# Plan: Catalog Management (dresses, variants, media)

**Created**: 2026-07-24 · **Status**: ready-to-build · **Spec**: [`../specs/catalog-management.md`](../specs/catalog-management.md) (rev 2, approved-for-build) · **Epic**: E2 Feature 8 · **Effort**: M
**Branch**: `feature/catalog-management` off `main`, in a worktree at `.worktrees/catalog-management`

Sequential TDD, red before green, five feature commits — one per task, never one per file. **No Docker locally**: the local loop is `pytest -m "not db"` + ruff + mypy under `uv`; anything needing Postgres or MinIO carries `pytest.mark.db` (and `s3` is *always* applied alongside `db`) and only goes green in CI. Each task below states exactly which assertions are locally verifiable and which wait for the pipeline. Money is integer agorot everywhere. Where this plan and the spec disagree, **the spec wins** — the two places I found it under-determined are called out inline as `SPEC GAP`. At build start: E2 epic table F8 → `building` + spec/plan links (E1 table convention).

Repo path contains a space and a `+`: quote every `cd` (`cd "$(CURDIR)/backend"`). Git tracks `backend/` and `frontend/` lowercase; the on-disk dirs are `Backend/` and `Frontend/`. Use the lowercase form in every path literal that ends up in code (notably the parity test's path to `validation.ts`) — macOS resolves it case-insensitively and Linux CI checks out lowercase.

## Why this order

**1 → 2**: the ORM row types are what the repositories return and what carries `storage_key`, and `test_every_tenant_id_table_has_forced_rls` (`tests/test_tenant_isolation.py:203`) turns *any* later table lacking FORCE RLS into a red isolation suite — so the three tables land with their grants and policies in one reviewable commit or every intermediate commit is red. **2 → 3**: `CatalogService.__init__` takes a `MediaStorage`; building the service first forces either a `None`-storage branch (exactly what `UnconfiguredMediaStorage` exists to eliminate) or a second refactor, and task 2 front-loads `uv lock` + the boto3 mypy override + the `s3` marker so every later push installs and typechecks against the final dependency set — reverse it and every task-3 push dies at `uv sync --locked` before a test runs. **3 → 4**: the router is a thin translator over the service's frozen views, `app/errors.py` must exist before `create_app()` can register handlers on the base classes, and the fake in `test_catalog_api.py` is typed against the real service signatures. **4 → 5**: `api.ts`'s wire types are transcribed from `schemas.py`, not from the spec, and `test_frontend_constant_parity.py` compares two files that must both already exist. Reordering 3↔4 leaves the whole catalog module answering **500** instead of 404/400 (the exact regression spec §Decisions-7 documents); reordering 2↔3 leaves a null-storage branch nobody deletes later.

**Gotchas.** **`record_failure` on success.** `FixedWindowRateLimiter` counts only what is explicitly recorded — its docstring says "successes never count" (`app/auth/rate_limit.py:9-11`) — so a *successful* presign, which authorises a 10 MiB write to our bucket, must call `record_failure(f"presign:{tenant_id}")` by hand. This is the `create_terms_version` precedent (`app/boutique/service.py:325`); keep the same explanatory comment beside it or the next reviewer deletes it as a bug, and the throttle silently becomes inert. **No `MediaStorage` network method inside `tenant_session`.** `tenant_session` is `async with session_factory() as session, session.begin()` (`app/db/tenant.py:25`) — the whole block is one Postgres transaction, and boto3's default 60 s connect + 60 s read timeouts would pin a pool connection *and* a per-dress advisory lock across an S3 stall. Concretely: presign = throttle check → session #1 (lock, resolve dress, sweep stale pendings capturing their keys, count, mint `media_id`, build key, `insert_pending`) → commit → **outside**: best-effort `delete_object` per swept key, `presigned_post`, `record_failure`. Confirm = session #1 (resolve, idempotent short-circuit on `ready`) → **outside**: `head_object` then `read_prefix(length=16)` → session #2 (lock, re-read, promote only `if still pending`, build the view). Media delete = lock → soft-delete → commit → `delete_object` outside (failure logged at WARNING with `tenant_id`/`dress_id`/`media_id`/`storage_key`, then swallowed — that log line is the future reconcile job's only input, so silence is not acceptable) → read transaction for the detail view. Signed URLs are minted after the session closes. **Two lock prefixes**, both `pg_advisory_xact_lock(hashtext('…:' || :dress_id))` with bound params, never f-strings: `dress-media:` and `dress-variants:` — distinct so photo work and stock work on one dress never serialise, and both clear of F7's un-prefixed `hashtext(:tenant_id)` space (`app/boutique/service.py:236-239`). **Resolve the parent dress first**: the first statement inside the session — and, for presign, inside the lock — is `dress = await dresses.by_id(session, tenant_id, dress_id)`, `if dress is None: raise CatalogNotFoundError`, with `by_id` carrying `tenant_id`, `id` **and** `deleted_at IS NULL` (the `appointment_types.by_id` predicate verbatim); `build_media_key` runs only after that returns a row, so no unverified id ever reaches an S3 key. Every media repository method takes `(session, tenant_id, dress_id, media_id)` and puts all three in the `WHERE` alongside `deleted_at IS NULL` — a media id belonging to a different dress *of the same tenant* is a 404, never a silent re-parent. `media_id` is minted client-side (`uuid.uuid4()`) and passed as the explicit `id` on the single INSERT alongside the pre-computed `storage_key`; there is **no** post-insert UPDATE of `storage_key` (`.memory/patterns/atomic-parent-child-across-rls.md`). `_stock_summary` is the single `out_of_stock` formula, fed by the pre-aggregated group row for lists and by loaded rows for the detail, never a per-row query; zero variants ⇒ `out_of_stock = True`. Archive is one transaction with `WHERE … AND deleted_at IS NULL` against `dress_media`, then `dress_variants`, then `dresses`, so `now()` (= `transaction_timestamp()`) stamps all three identically and individually-deleted children keep their older stamp; restore reads `dresses.deleted_at` **first** (404 if already NULL) and matches children on `deleted_at = :archived_at`. The service clamps `offset`/`limit` again after the router does (F7 precedent) so a non-router caller cannot request an unbounded page. `IntegrityError` on the `lower(size_label)` index → `DuplicateSizeError`, never a raw 500; the service also normalises and rejects in-payload duplicates before touching the DB. `MediaNotConfiguredError` / `MediaStorageUnavailableError` are **imported from `app/storage/`**, never redefined here. The re-parenting in this commit is behaviour-neutral — Starlette resolves handlers by `type(exc).__mro__`, so the existing subclass registrations still match; task 4 moves them to the bases.

**Done.** Local: `cd backend && uv run pytest -m "not db" -q` (only `test_catalog_validation.py` is new here), `uv run ruff check .`, `uv run ruff format --check .`, `uv run mypy app tests`, plus `test_boutique_*` still green after the re-parenting. CI-only: everything in `test_catalog_integration.py`, `test_catalog_isolation.py` and the two new `test_media_upload_s3.py` rows — every concurrency, lock, sweep and statement-count assertion in this task is unverifiable on the local loop.

## Risks during build

1. **The `app/schemas.py` hoist ripples.** It touches `app/boutique/schemas.py` and `app/boutique/router.py`'s import list, and a bare re-export trips ruff `F401`. Re-run `test_boutique_api.py` (which asserts `{"ok": true}` bodies) immediately after the move, not at the end of task 4.
2. **Tasks 3 and 4 are not independently shippable.** Between them the catalog error classes exist but no handler is registered on the base classes, so a catalog 404 would be a 500. That is intentional and bounded, but do not open a PR containing only task 3.
3. **`uv lock` and the repo path.** Always `cd "…/backend"` with quotes (space + `+` in the path). The lockfile must be staged with `pyproject.toml` in commit 2 or every subsequent CI run dies at `uv sync --locked` before a test executes.
4. **MinIO CI budget.** ~45–75 s added to the backend job, cold image pull dominating. If the job grows by more than ~2 min, split `s3` into its own CI step rather than loosening any assertion. Never request `minio_container` at module scope — a module-level boto3 client starts the container during collection even under `-m "not db"`.
5. **The webp magic prefix is not a `startswith`.** `RIFF` sits at offset 0 and `WEBP` at offset 8; a single-prefix table passes jpeg and png and then silently accepts anything RIFF-shaped. Read 16 bytes and assert both segments.
6. **`record_failure` on a success reads like a bug.** It will be deleted in review unless the F7-style comment travels with it. Same for the `is_blocked` pre-check running *before* any work in the presign path.
7. **The statement-count assertion is brittle by default.** `tenant_session` issues a `set_config` statement of its own, so a naive `before_cursor_execute` counter sees five, not four. Decide the filter (count only statements touching `dresses`/`dress_variants`/`dress_media`) before writing the test, not after it flakes.
8. **Advisory-lock keys.** `pg_advisory_xact_lock(hashtext(...))` must be executed via `text()` with bound parameters, never interpolation. A `hashtext` collision between two dresses is acceptable (a brief false share); a *missing* prefix is not — it would collide with F7's tenant-keyed lock space and serialise unrelated settings writes.
9. **`make lint` gets slower and stricter** once its backend line typechecks `tests` too. CI already ran `mypy app tests`, so expect zero new errors — but the local loop now surfaces them before push instead of after.
10. **Two `SPEC GAP` resolutions above** (`InMemoryMediaStorage` location, `ForbidExtraModel` hoist) are the plan's calls, not the spec's. If review disagrees, the spec wins and the plan is amended.
